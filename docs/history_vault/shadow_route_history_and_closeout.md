# Shadow Route History and Closeout

**Status:** historical diagnostic route / HOLD  
**Closed in:** `v1.6.13-alpha`  
**Native witness affected:** no  
**Discovery claim earned:** no

## Why this document exists

The v1.6 shadow line produced useful engineering and a stronger testing form, but it did not earn the scientific claim it was built to test.

The README and ROADMAP should not carry shadow-test visuals, result tables, or active-route pressure. Those public surfaces should keep the main native evidence route clear. This file preserves the shadow route as history so the work is neither hidden nor mistaken for success.

## Claim separation

Native ZeroGateSim asks whether the final trinary witness can refuse false crowns inside controlled synthetic-field evidence.

The shadow route asked a harder side-question:

> Can a role-stripped report-side score infer false-one-like pressure from observable behavior without designed role labels, and beat simple baselines?

Current answer:

```text
native four-gate witness: standing
role-stripped shadow density pressure: partially visible
relation-specific shadow discrimination: not earned
return-specific shadow discrimination: not earned
demotion-specific shadow discrimination: not earned
role-blind discovery: not earned
```

## What was useful

The shadow route was not empty work. It gave the repo stronger testing machinery:

- role-stripped feature / target separation;
- forbidden-field audits;
- score-before-target discipline;
- baseline and falsifier comparison;
- triad27 preflight;
- weather hardening across `triad27`, `deep81`, and `wide243` surfaces;
- hardened cell-level triad27 evidence generation;
- residual discrimination diagnostics;
- lane-specific candidate scores;
- runs inventory classification.

That machinery remains useful as witness infrastructure. The failed claim is not the same thing as failed learning.

## What failed

The hardened triad27 route showed that shadow scoring could track pressure density, but did not beat simple baselines strongly enough on the pressure-kind questions that matter most:

- raw false-one specificity;
- demotion specificity;
- hold-or-demote pressure;
- relation-specific false pressure;
- return-specific false pressure.

The key wound:

```text
pressure amount != false-one kind
```

Raw pressure and raw strength remained too strong as dumb baselines. The shadow route did not show enough independent signal after those baselines were accounted for.

## What remains allowed

Allowed:

- cite the shadow route as a failed / held diagnostic exploration;
- keep the tooling available for future hypotheses;
- use its hardening reports to prevent easy baseline mistakes;
- use it as evidence that the repo can falsify its own attractive routes.

Forbidden:

- role-blind discovery language;
- detector closeout language;
- deep81 / wide243 shadow trust based on the failed triad27 specificity route;
- pretending the v1.6 shadow route proved more than it did;
- deleting the failed path from history to make the repo look cleaner.

## Future return condition

The shadow route may return only if a genuinely new hypothesis is proposed. A new hypothesis must state before coding:

- what signal it expects beyond raw pressure / raw strength;
- which target it should beat;
- which simple baselines would falsify it;
- what result would stop the route again.

No more one-more-feature drift.
