# Controlled Synthetic-Field Language Boundary

**Line:** `v1.5.3-alpha`  
**Status:** language boundary for the v1.5 experiment line

ZeroGateSim started with a first-research-alpha proof record described as generated toy-field evidence. That language remains correct for the historical proof floor and for manuscript lineage.

The v1.5 line is now allowed to use a stronger phrase in a narrower place:

```text
controlled synthetic field
```

This is not cosmology validation, physical dimensional-genesis validation, or proof that reality itself is trinary. It is a claim about experiment shape: generated fields are now explicit, seeded, parameterized, adversarial, and reported through reproducible matrix bundles.

## Language ladder

Use this ladder in public docs and paper-two notes:

```text
generated toy field
-> controlled synthetic field
-> adversarial synthetic benchmark
-> externally reproducible modeling framework
-> empirical/model-comparison bridge
```

Current earned step:

```text
controlled synthetic field
```

Not yet earned:

```text
adversarial synthetic benchmark
```

That later term requires role-blind shadow testing, independent generator families, baseline comparisons, and external reproduction.

## What remains toy-field language

Keep `toy field` or `generated toy field` when referring to:

- the first-research-alpha proof record;
- the original simulation-supported manuscript boundary;
- historical proof-card numbers;
- early proof-harness language;
- any claim that could otherwise sound like physics validation.

Correct:

```text
The first-research-alpha proof record remains a generated toy-field proof-of-concept.
```

Incorrect:

```text
The first-research-alpha proof record proves controlled physical dimensional emergence.
```

## What may now use controlled synthetic-field language

Use `controlled synthetic field` for v1.5+ reports when the run has:

- explicit candidate profile;
- explicit matrix profile;
- explicit seed range;
- explicit step count;
- explicit gate and strength thresholds;
- dedicated four-gate adversary coverage when claimed;
- result files bundled with strict handoff include rules;
- claim boundaries that refuse cosmology and role-blind overclaim.

Correct:

```text
The v1.5 seed-block report reads completed controlled synthetic-field evidence across distinction, polarity, relation, and return adversary corpora.
```

Incorrect:

```text
The controlled synthetic field proves that real dimensions are generated by ZeroGateSim.
```

## The 27 / 81 / 243 weather ladder

The matrix counts are not decorative and not decimal habit. They are full-factorial trinary weather designs.

Each weather axis has three levels:

```text
-1 = reduced / resistant / calmer pressure
 0 = baseline / witness pressure
+1 = amplified / expansion pressure
```

### triad27

```text
triad27 = 3^3 = 27 scenarios
```

Axes:

- noise pressure;
- relation pressure;
- expansion pressure.

Purpose:

- fast controlled smoke and seed-block evidence;
- basic four-gate adversarial weather;
- enough structure to see whether pressure appears without running the heavy proof weather every patch.

### deep81

```text
deep81 = 3^4 = 81 scenarios
```

Adds:

- perturbation pressure.

Purpose:

- field-kick / late-shock testing;
- checks whether a candidate remains coherent after disturbance;
- useful before changing gates because it can reveal whether a wound is caused by perturbation rather than by the native witness law.

### wide243

```text
wide243 = 3^5 = 243 scenarios
```

Adds:

- temporal-depth / time-axis pressure.

Purpose:

- broad current proof weather;
- temporal cohesion and return-depth stress;
- tests whether earned-one and false-one demotion survive compressed, baseline, and stretched temporal conditions.

`wide243` is the slow, space-hungry weather. It should not be run for every small patch. It is scientifically useful when the question is robustness under the full current weather grammar.

## How to use existing large local runs

Large PC-side runs are useful when they include reports or bundles that can be inspected without importing raw generated clutter into Git.

Best upload targets for review:

- `matrix_bundle.zip` from an individual matrix;
- `cross_logic_report_bundle.zip`;
- `seed_block_report_bundle.zip`;
- `threshold_sensitivity_bundle.zip`;
- `witness_ablation_bundle.zip`;
- a strict assistant handoff ZIP that includes the above read files and summaries.

Do not add raw `runs/` output to Git. Keep generated evidence local, uploaded for review, or archived externally when a DOI/evidence-release plan exists.

## Paper-two relevance

Paper two should be able to say:

```text
The project moved from generated toy-field proof-of-concept language into controlled synthetic-field evidence only after four-gate adversary coverage, seed-block reporting, threshold sensitivity reporting, and witness ablation accounting existed.
```

That sentence is stronger than calling everything toy forever, but it still refuses the forbidden leap to physical proof.

## Boundary

Controlled synthetic-field language means:

```text
controlled generator + trinary weather matrix + seeds + thresholds + adversarial candidate profile + report bundle + falsifier boundary
```

It does not mean:

```text
physics proof
cosmology proof
role-blind discovery
external empirical validation
identity with known formal logics
```
